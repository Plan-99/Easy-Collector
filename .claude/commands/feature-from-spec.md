argument-hint: "<spec-path>  (예: docs/queue/01_add_filter.md)"

자율 운영의 1 사이클. **하나의 spec을 받아 worktree에서 구현·검증·PR까지** 진행합니다.
`/overnight-queue`가 spec마다 이 명령을 호출합니다.

인자: `$ARGUMENTS` — spec 파일 경로 (필수).

---

## 워크플로우

상위 컨텍스트는 spec 1개에 대해서만 책임을 짐. **다른 spec, 다른 worktree, 다른 브랜치는 절대 건드리지 않음.**

### 0. 환경 확인 & spec 파싱

```bash
test -f "$ARGUMENTS" || { echo "[feature-from-spec] spec 없음: $ARGUMENTS"; exit 64; }
```

spec 파일을 Read로 읽어 frontmatter(`size`, `verify`, `affected`, `prerequisite_specs`)와
acceptance criteria, 검증 hooks 블록을 추출. 추출한 정보는 사이클 끝까지 메모리에서 사용.

`prerequisite_specs`가 있는데 그 spec들이 `docs/queue/_done/`에 없으면:
- `--status skip` + `--note "prereq missing"` 로 보고 후 종료.

### 1. Baseline 스모크

`bash scripts/smoke.sh` 호출. exit ≠ 0이면 **인프라 문제**:
- 어떤 검사가 실패했는지 보고
- 종료 코드 2면 backend 컨테이너 자체가 없음 — `docker compose up -d backend ros2`를 시도하고 30초 대기 후 재호출
- 그래도 실패면 `--status fail --stage baseline-smoke` 로 보고 후 종료. **이 spec은 건드리지 않음.**

### 2. Worktree 생성

브랜치 이름: `overnight/<slug>` (slug = spec 파일명에서 `NN_` prefix와 `.md` 제거).
이미 같은 이름의 브랜치/worktree가 있으면 `-2`, `-3` 등으로 접미사 붙임.

```bash
BASE="$(git rev-parse --show-toplevel)"   # base 체크아웃 절대경로 (host 무관, 아래 spec 이동에서 재사용)
SLUG="$(basename "$ARGUMENTS" .md | sed 's/^[0-9]*_//')"
BRANCH="overnight/$SLUG"
WT="/tmp/easytrainer-wt/$SLUG"
mkdir -p /tmp/easytrainer-wt
git worktree add "$WT" -b "$BRANCH" HEAD
```

이후 모든 작업은 `cd "$WT"` 안에서 수행. `$WT` 밖의 파일을 절대 손대지 않음.

### 3. 구현

spec의 "변경 내용 (What)" + "영향 파일" 섹션을 기준으로 코드 수정.

- **모호한 결정은 보수적으로**: spec에 명시 안 된 사이드 변경 금지. "ㅇㅇ도 같이 고치는 게 좋겠다" 같은 판단은 **하지 않음** (PR 리뷰에서 코멘트로 해결).
- spec의 "비-목표 (Out of Scope)"에 적힌 건 절대 손대지 않음.
- 새 파일 생성은 spec의 "영향 파일" 범위 안에서만.

각 의미 있는 단계마다 `cd "$WT" && bash scripts/quick_apply.sh "$WT" /opt/easytrainer/project`
**호출하지 않음** — quick_apply는 dev checkout → 런타임 동기화용. 자율 운영에서는 worktree
자체를 검증 대상으로 보지 않고 **머지 후 main에서 검증**됨. 자율 사이클의 검증은 *코드 변경
자체가 일관적으로 적용됐는가*가 핵심.

다만 **검증 단계에서 런타임 endpoint를 hit해야 하는 경우** (Acceptance에 `curl ...`이 있는 경우)
quick_apply 1회 실행 후 검증. 검증 후엔 머지 전 상태로 되돌릴 필요는 없음 (다음 사이클이
또 quick_apply 할 것이므로).

### 4. 검증

3단 게이트 — spec frontmatter의 `verify:` 값에 따라.

| `verify` | 단계 |
|----------|------|
| `none`   | smoke만 |
| `smoke`  | smoke + spec의 "검증 hooks" 블록 모든 명령 |
| `e2e`    | smoke + verify hooks + `python3 -m backend.tests.test_e2e_pipeline --skip-train` |

각 단계 실패 시:
- **1회 한정 재시도**: 실패 출력을 읽고 구현 단계로 돌아가 1회만 수정 후 재검증.
- 두 번째 실패면 `--status fail --stage verify` 로 보고 + worktree 보존 + 종료.

### 4.5. 매뉴얼 자동 동기화

검증을 통과한 변경의 파일 목록이 **사용자 매뉴얼 영향 패턴**과 매칭되면 `manual-writer`
서브에이전트를 호출해 `home-next/src/app/docs/_content/ko/**/*.mdx` 를 같이 갱신.
이렇게 갱신된 매뉴얼 변경은 **같은 PR에 포함**시킨다 (검토 비용 절감).

**트리거 패턴** (한 줄이라도 매칭하면 시도):

- `frontend/src/pages/v2/**`
- `frontend/src/components/v2/**`
- `release/ui/**`
- `modules/*/module.json`
- `backend/api/routes/**`
- `README.md`

**스킵 조건**:

- spec frontmatter `manual: skip` (내부 리팩터 등 사용자 노출 없음)
- spec frontmatter `manual: already` (spec 작성자가 같은 변경에 매뉴얼도 직접 적은 경우)
- 매칭되는 변경 파일 0개

```bash
# 변경된 파일 수집 (worktree 안)
cd "$WT"
CHANGED=$(git diff --name-only "$(git merge-base HEAD main)..HEAD")
MATCHED=$(echo "$CHANGED" | grep -E \
  '^(frontend/src/(pages|components)/v2/|release/ui/|modules/[^/]+/module\.json$|backend/api/routes/|README\.md$)' \
  | tr '\n' ',' | sed 's/,$//')
```

`MATCHED`가 비어있지 않고 spec의 `manual`이 `skip`/`already`가 아니면:

```
Agent({
  description: "Sync manual for autonomous cycle",
  subagent_type: "manual-writer",
  prompt: "다음 변경 파일들에 대해 매뉴얼을 갱신합니다. 작업 디렉토리: <$WT> (이 worktree
           안에서만 편집, base 체크아웃은 건드리지 말 것). 매뉴얼 편집은 home-next/src/app/docs/_content/ko/
           안에서만. 커밋/푸시는 하지 말 것 — 호출자가 모아서 커밋합니다.

           files=<MATCHED>

           lint/tsc 가 home-next 의 기존 에러로 실패하면 본인이 만든 변경분만 통과하는지
           확인 후 본인 변경분에 lint 에러 없으면 통과로 보고."
})
```

**실패 처리** (논블로커):

- 서브에이전트가 에러나 빈 결과를 반환하면 **사이클 전체를 fail 처리하지 않음**.
- summary 의 `note`에 `manual: skipped (<사유>)` 추가.
- 코드만 들어간 PR 로 진행.

**성공 처리**:

- worktree 안의 `home-next/src/app/docs/_content/ko/` 변경이 unstaged 상태로 남아 있어야 함
  (서브에이전트가 commit 안 함). 다음 단계 `git add -A` 가 코드 변경과 함께 묶어 커밋.
- PR 본문에 `Manual: <N>개 페이지 갱신` 줄 추가.

### 5. 커밋 & PR

검증 + 매뉴얼 동기화 통과 시 worktree 안에서:

```bash
cd "$WT"
# 매뉴얼 변경 행 수 (PR 본문용)
MANUAL_FILES=$(git diff --name-only -- 'home-next/src/app/docs/_content/ko/**/*.mdx' | wc -l)

git add -A
git commit -m "$(printf '%s\n\n%s' "<spec title>" "spec: $ARGUMENTS

🤖 generated by /feature-from-spec
")"
git push -u origin "$BRANCH"
gh pr create --base main --head "$BRANCH" \
  --title "<spec title>" \
  --body "$(cat <<EOF
## Summary

<spec의 '목표' 섹션을 1–3 문장으로>

## Changes

$(git diff --stat main..."$BRANCH")

## Acceptance

<spec의 acceptance criteria를 그대로 옮기되, 검증된 항목은 [x]>

## Manual

$( [ "$MANUAL_FILES" -gt 0 ] && echo "갱신된 매뉴얼 페이지: $MANUAL_FILES" || echo "해당 없음 (사용자 노출 변경 없음 또는 manual: skip)" )

## Spec

\`$ARGUMENTS\`

🤖 generated by /feature-from-spec
EOF
)"
```

PR URL을 캡처해서 보고에 포함. `note`에는 매뉴얼 동기화 결과도 포함 (예: `manual: 2 pages` 또는 `manual: skipped`).

### 6. spec 이동 (성공 시)

```bash
git -C "$BASE" mv "$ARGUMENTS" "docs/queue/_done/$(basename "$ARGUMENTS")"
# 이 mv는 base 체크아웃에서 직접 — worktree 안에서 하지 않음 (queue 자체는 base의 자산)
```

base 체크아웃에서 mv를 하면 다른 worktree에서 같은 spec을 픽업하는 일을 방지할 수 있음.
단 base에 uncommitted 변경이 있으면 mv가 실패할 수 있으니 시도만 하고 실패는 warn으로 처리.

### 6'. 실패 시 spec 이동

`_failed/` 로 mv. 실패 stage를 spec 파일 맨 위 한 줄 주석으로 박음:

```bash
git -C "$BASE" mv "$ARGUMENTS" "docs/queue/_failed/$(basename "$ARGUMENTS")"
# spec 본문 맨 위에 "<!-- failed: stage=verify at <ts> -->" 추가
```

### 7. 보고

상위 호출자에게 (또는 `overnight_summary.sh cycle` 인자로) 다음 정보 반환:

- spec path
- status: `pass` | `fail` | `skip`
- stage (fail일 때): `baseline-smoke` | `implement` | `verify` | `commit` | `pr`
- branch
- PR URL (있을 때)
- elapsed seconds
- 한 줄 note

## 안전 가드

- **base 브랜치 절대 직접 수정 금지**. 모든 변경은 worktree.
- `git push --force`, `git reset --hard`, `git clean -fd` 사용 금지.
- spec이 "Out of Scope"로 명시한 폴더 수정 금지.
- worktree가 5GB 이상 disk를 사용하면 중단 (자가 점검: `du -sh "$WT"`).
- 한 사이클이 90분을 넘기면 자체 중단 (실패로 분류).

## 디버깅 / 매뉴얼 실행

자율 운영이 아닌 *수동*으로 spec 하나 처리하고 싶을 때도 그대로 사용 가능:

```
/feature-from-spec docs/queue/01_my_feature.md
```

이 경우 worktree는 보존되며 사용자가 직접 `cd`해서 확인 가능.
