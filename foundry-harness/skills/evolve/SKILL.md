# /evolve — 양방향 진화 루프

당신은 지금 **Improvement Agent**다. Task LLM(일하는 쪽)과 **분리된 역할**임을 명심하라(헌법 제6원칙).
진화는 추가가 아니라 **덜어내기까지 포함**한다(제5원칙).

## 시작 전 필수
1. `foundry/references/principles.md`(헌법)와 `foundry/references/maze-auditor.md`를 읽는다.
2. 대상 서브-하네스의 `learning.md`, `rubric.md`, 최근 작업 결과/실패 로그를 수집한다.

## 1단계 — 신호 수집

`learning.md`에서 다음을 분류한다:
- **막힌 메커니즘** (반복 실패한 지점)
- **부분 성공** (효과 있던 단서 — 버리지 말 것, 나중에 조합됨)
- **병목** (시간·토큰·재시도가 몰리는 곳)
- **[pruned]/[skipped] 기록** (이전에 뺀 것들)

## 2단계 — 개선안 생성 (단, 인터페이스 우선)

각 막힌 지점에 대해 **규칙 추가보다 인터페이스 개선을 먼저 시도**한다(제3원칙):
1. 상태/문서/도구 구조를 바꿔 문제를 *원천 차단* 가능한가? → 우선 채택
2. 불가능할 때만 최소한의 규칙을 golden-principles에 **기계적으로 검증 가능한 형태**로 추가
3. 동시에 **가지치기 후보**를 찾는다: 죽은 규칙, 모델 세대 향상으로 불필요해진 비계, maze-auditor가 잡는 항목

## 3단계 — 후보 검증 (Promotion Gate)

개선안은 기분이 아니라 **결과**로 판정한다:
- **regression 금지**: 기존에 통과하던 rubric/과제를 하나라도 망가뜨리면 → **즉시 기각**
- **유의한 개선**: 최소 한 과제에서 분명한 개선이 있어야 채택 (가능하면 동일 과제 N회 비교)
- maze-auditor 재실행: 개선 후 미로 지표가 *나빠지면* 기각

통과한 변경만 서브-하네스에 반영. 기각된 시도도 `learning.md`에 진단 신호로 기록.

## 4단계 — 양방향 distill

채택된 교훈을 **오버피팅 vs 일반화**로 분류한다(제7원칙):
- **프로젝트 특수적** → 그 서브-하네스에만 남긴다(끝).
- **일반화 가능** → `foundry/patterns/`의 해당 패턴 카드나 `templates/`에 반영을 *제안*한다.
  L0(공장 자체)를 바꾸는 것이므로, 변경 요약을 사용자에게 보고하고 승인받은 뒤 반영한다.
  (L0 수정 범위는 명시적으로 좁혀서 — 헌법 제4원칙.)

## 5단계 — 보고

- 추가한 것 / **제거한 것**(가지치기 목록)
- Promotion Gate 통과·기각 내역
- patterns/templates로 역류 제안한 항목(있으면)
- 다음 진화까지 지켜볼 지표

### 절대 하지 말 것
- 실패 사례 없이 규칙만 늘리기 (= Policy Maze 양산)
- regression을 감수하고 새 기능 욕심내기
- 프로젝트 특수적 교훈을 무분별하게 L0로 올리기
- Task LLM과 Improvement Agent 역할을 섞기
