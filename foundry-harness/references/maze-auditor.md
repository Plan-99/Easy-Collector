# Maze Auditor — Policy Maze 자가 감사 체크리스트

> 주조(forge) 5단계와 진화(evolve)에서 사용한다.
> 목적: 갓 만든 서브-하네스가 "모델이 본업보다 *규칙 푸는 데* 더 헤매는" 미로가 아닌지 검사.
>
> **출처**: *개념*("Policy Maze")은 **Henry Pan**, Self-Improvement Harness —
> *"the Task LLM is no longer solving the original task, it is now navigating a harness policy maze."*
> *7개 체크리스트 항목·"2개 이상이면 거부" 판정선·가지치기 우선순위*는 그 진단을 실행 가능하게 만든
> **Foundry의 조작화(operationalization)** 이며 원문에 그대로 있는 것은 아니다.

## 측정 질문 (각 항목 yes/no)

1. **분량**: golden-principles + AGENTS.md 합이 모델이 매 작업마다 읽어야 하는데도 과도하게 긴가? (목표: AGENTS.md ≤ ~100줄, golden-principles는 "기계적으로 검증 가능한" 규칙만)
2. **선택 부담**: agent가 너무 많아 "어떤 agent를 부를지" 자체가 과제가 되는가? (경험칙: 도메인당 핵심 역할 3~6개로 시작)
3. **규칙 충돌**: 서로 모순되거나 우선순위가 불명확한 규칙이 있는가?
4. **죽은 규칙**: 실제 실패 사례 없이 "혹시 몰라서" 넣은 규칙이 있는가? (→ 헌법 제1원칙 위반)
5. **검증 불가 규칙**: "좋은 코드를 짜라" 같이 *기계적으로 판정 불가능한* 훈계가 섞였는가?
6. **인터페이스로 대체 가능**: 규칙으로 막은 것 중, 도구/상태 구조 변경으로 *원천 차단* 가능한 게 있는가? (→ 제3원칙)
7. **본업 비율**: dry-run에서 Task LLM이 쓴 토큰/턴 중, "규칙 해석·충족"이 "실제 과제 해결"보다 큰 비중인가?

## 판정

- 1~7 중 **2개 이상 yes → 주조 거부**, 4단계로 회송하여 **가지치기(pruning)** 후 재검.
- 가지치기 우선순위: ④죽은 규칙 → ⑤검증 불가 규칙 → ⑥인터페이스 대체 가능 → ②선택 부담.

## 가지치기 기록

제거한 요소는 버리지 말고 대상 프로젝트의 `learning.md`에 한 줄로 남긴다:
```
- [pruned] <요소> — 이유: <감사 항목 번호/사유>
```
나중에 정말 필요해지면 근거와 함께 되살린다.

## 모델 세대 점검 (model-generation audit)

> 위 7문항은 **제1원칙(Policy Maze=규칙 과잉)** 만 본다. 이 섹션은 **제5원칙(모델 세대가 올라가면 옛 모델용 비계는 짐이 된다)** 을 조작화한다.
> **언제 실행:** 런타임 모델 세대가 바뀐 직후의 `/evolve`(업그레이드 트리거). 평상시 evolve 에선 생략 가능.
>
> **출처:** *원칙*은 **Anthropic**, Harness Design — "Evolution with Model Capability"(모델이 좋아지면 load-bearing 컴포넌트가 불필요한 오버헤드가 된다) + **Henry Pan** — 약한 모델용 schema 규칙이 강한 모델(Sonnet 4.6) 성능을 *깎았다*("model-specific tax").
> *체크리스트화·"불변식 vs 비계" 판정선*은 Foundry 의 조작화다. 근거 실행: coin-model 4.8 evolve.

각 하네스 요소(규칙·agent·문서 비계)마다 한 줄로 답한다:

1. **무엇을 보상하나**: 이 요소가 *대신 메우는 모델 약점*은 무엇인가? (예: "긴 작업서 일관성 붕괴", "자기 결과 후한 채점", "진척 보고 누락")
2. **불변식인가 비계인가**:
   - **환경/도메인 불변식** — 모델과 무관하게 참(수수료·look-ahead 금지·회귀 게이트·비밀 분리·실행-경로 parity). → **세대와 무관, 유지(keep).**
   - **모델 보상용 비계** — 특정 모델 약점을 메우려 넣음(강제 진척보고·sprint 분해 크러치·액션 제한 schema·verbose 잔소리). → 3번으로.
3. **약점이 해소됐나**: 그 약점이 *현 모델*에서 해소(또는 충분히 완화)됐는가? (모델 릴리스 노트/마이그레이션 가이드로 *근거*를 댄다 — 기분 금지.)
   - 해소됨 → **prune 후보**(아래 가지치기 기록으로).
   - 미해소 → 유지하되, 근거를 적어 다음 세대에 재점검.

### 판정
- "보상용 비계"이면서 "약점 해소됨"인 요소는 **prune 후보**다. 단 실제 제거는 **Promotion Gate** 를 거친다(통과하던 test/과제를 깨면 기각 — 제거도 회귀를 일으킬 수 있다).
- 모든 요소가 *불변식*으로 분류되면 가지치기 0이 정상이다(요소가 모델 약점이 아니라 도메인 진실을 인코딩했다는 뜻). **억지 삭제 금지** — 그것도 자기 작업 부풀리기(제6원칙)다.

> **범위 밖(별도 강제):** 이 체크리스트는 *Policy Maze*(규칙 과잉)와 *모델 세대 비계*를 본다. **검증 환경 parity**

> 는 성격이 다르므로 여기 섞지 않고 **forge 5단계(validation)** 에서 강제한다. 근거 실패: coin-model F1.
